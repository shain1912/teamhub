# AI 비서 E2E 검증

TeamHub의 인앱 AI 비서는 비결정적 LLM(GLM)이 도구를 호출해 티켓·스프린트·간트·체크리스트 등을 만든다.
LLM은 같은 작업을 여러 번 호출하거나(중복 생성), 여러 항목 중 일부만 호출(누락)하기 쉽다.
**유닛 테스트로는 못 잡는다 — 실제 모델을 호출하는 E2E 재현만이 잡는다.**

## 언제 쓰나 (Why)

`src/lib/aiTools.ts`(도구·실행기), `src/components/AiChat.tsx`(에이전트 루프·시스템 프롬프트),
`src/lib/glm.ts`, 서버 `server/src/index.ts`의 `/ai/chat`를 수정하면 모델의 도구호출 패턴이 바뀐다.
사용자가 "여러번 생성된다 / 하나만 생긴다 / 이상하게 만든다"고 하면 거의 항상 이 계층의 문제다.

## 핵심 방법론

1. **임시 사용자로 진짜 토큰을 만든다.** `/ai/chat`는 Supabase 로그인 토큰을 요구한다.
   admin API로 임시 유저를 만들고 password grant로 토큰을 받은 뒤, **끝나면 즉시 삭제**한다.
   (이게 가능한 이유: 우리는 service_role 키를 갖고 있고, 임시 유저는 example.com 으로 명확히 표시 후 정리)
2. **에이전트 루프를 앱과 동일하게 재현한다.** 시스템 프롬프트 + 도구 스키마를 프록시에 보내고,
   돌아온 tool_calls를 **모의 실행기**로 처리한 뒤 결과를 다시 모델에 넣는 루프를 돈다.
   DB는 건드리지 않는다 — 실행기는 멱등·상태형 모의(mock)다.
3. **간헐 버그를 위해 반복한다.** LLM은 비결정적이라 1회로는 못 잡는다. 같은 시나리오를 N회(기본 5) 돌린다.
4. **중복과 누락을 동시에 본다.** 도구호출 시퀀스를 기록하고, 최종 생성 개수가 기대와 다르면 실패로 판정한다.

## 실행

```bash
node .claude/skills/ai-agent-verify/scripts/ai_e2e.mjs            # 내장 시나리오 × 5회
node .claude/skills/ai-agent-verify/scripts/ai_e2e.mjs --runs 8  # 간헐 버그 의심 시 더 많이
```

- `.env`에서 `SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY`/`SUPABASE_SERVICE_ROLE_KEY`를 읽는다.
- **종료코드 0 = 통과, 1 = 버그 발견** → CI 게이트로 그대로 쓸 수 있다.
- 새 도구를 추가했으면 스크립트의 `T`(도구 스키마), `makeMock`(모의 실행기), `SCENARIOS`에 케이스를 추가한다.

## 버그를 발견하면 — 모델을 탓하지 말고 코드로 막아라

LLM 동작은 프롬프트로 "유도"할 수는 있어도 "보장"할 수 없다. 그래서 방어는 **실행기와 루프**에 둔다.
이번 프로젝트에서 실제로 적용한 3중 방어(재발 방지의 표준 패턴):

| 버그 | 원인 | 막는 법 (코드) |
|------|------|---------------|
| 중복 프로젝트/채널 생성 | `create_project`/`create_channel`에 멱등성 없음 | 같은 이름 있으면 insert 대신 **재사용**(`idExact`로 정확매칭 확인) |
| 간트 작업 fail→retry 노이즈/중복 | 없는 프로젝트로 첫 호출 실패 후 재시도 | `create_gantt_task`가 **프로젝트 자동 생성**(get-or-create) |
| 모델이 같은 생성을 반복 호출 | 비결정적 재호출 | 에이전트 루프에 **턴단위 dedup**: `name+JSON.stringify(args)` 서명을 Set으로 추적, 변경계열(create_/delete_/post_/add_/assign_/set_/move_/toggle_) 중복 시 실행 스킵 |
| 여러 항목 중 일부 누락 | 모델이 일부만 호출 | 시스템 프롬프트에 "항목마다 정확히 한 번씩 전부 생성" + 루프가 후속 호출 허용(최대 8회) |

원칙: **프롬프트는 유도, 실행기·루프는 보장.** 멱등성과 dedup으로 "여러 번 호출해도 한 번만 반영"되게 만들면 모델이 실수해도 데이터가 망가지지 않는다.

## 삭제 도구 주의

delete 계열은 **정확매칭(`idExact`)만** 허용한다 — 부분일치(ilike)로 지우면 엉뚱한 걸 지운다.
모호하면(`__AMBIGUOUS__`) 거부한다. 검증 시 delete 시나리오는 별도 임시 데이터로만 테스트한다.
